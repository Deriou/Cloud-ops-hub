# Cloud-Ops-Hub TDD 执行文档

## 1. 文档目标

本文件用于指导 Cloud-Ops-Hub 在业务实现过程中的测试先行（TDD）实践，确保：

- 每个实现步骤独立、可验证、可回滚
- 核心业务严格遵循 Red -> Green -> Refactor
- 在 8GB 单机资源约束下保持代码轻量、稳定与可观测

## 2. TDD 总流程（统一适用于所有模块）

每个功能点都按照以下步骤执行，不允许跳步：

1. 定义需求切片（一个切片只做一件事）
2. 编写失败测试（Red）
3. 最小实现使测试通过（Green）
4. 重构并保持测试通过（Refactor）
5. 执行回归测试 + 记录结果
6. 更新接口文档与运行手册

## 3. 测试分层策略

- 单元测试（必须）：
  - 关键算法、鉴权规则、模式切换、诊断聚合逻辑
- 集成测试（必须）：
  - Controller -> Service -> Repository 的主链路
  - 关键外部依赖交互（Jenkins/K8s/Prometheus/Loki）使用 Mock 或 Test Double
- 端到端测试（建议）：
  - 网关入口读写权限与演示模式限制
  - 关键用户路径（查看状态、触发诊断、查看报告）

## 4. 测试与实现的阶段划分

---

## 阶段 A：common 规范底座（必须先完成）

### A1. 统一返回体 `ApiResponse<T>`

实现要求（独立）：

- 定义标准字段：`code`、`message`、`data`、`traceId`
- Controller 统一返回 `ApiResponse<T>`

测试先行步骤：

1. 写测试：验证成功/失败响应结构完整
2. 运行测试（应失败）
3. 实现 `ApiResponse<T>`
4. 测试通过后重构工厂方法（如 `success()` / `fail()`）

验收标准：

- 任意接口响应字段完整且类型一致

### A2. 全局异常处理

实现要求（独立）：

- 定义 `BizException`
- 定义 `GlobalExceptionHandler`
- 禁止堆栈信息直接返回前端

测试先行步骤：

1. 写测试：抛出 `BizException` 返回业务码
2. 写测试：未知异常返回统一系统错误码
3. 最小实现异常映射
4. 运行回归测试，确认不泄露内部异常细节

验收标准：

- 异常输出结构与 `ApiResponse` 完全一致

### A3. 鉴权拦截器（核心）

实现要求（独立）：

- `X-Ops-Key` 双令牌机制
- Master Key 全权限
- Guest Key 仅允许 GET 且支持 TTL

测试先行步骤（重点）：

1. 写测试：无 key -> `401`
2. 写测试：Guest + GET -> 放行
3. 写测试：Guest + POST/PUT/DELETE -> `403`
4. 写测试：过期 Guest Token -> `401`
5. 实现 `AuthInterceptor` + Caffeine TTL
6. 全量鉴权测试通过后再做代码整理

验收标准：

- 鉴权规则与模式切换无歧义

## 阶段 B：Gateway-Portal（唯一公网入口）

### B1. 应用注册表查询

实现要求（独立）：

- 返回子应用元信息（名称、路由、状态）

测试先行步骤：

1. 写测试：空注册表返回空列表
2. 写测试：存在应用时返回正确字段
3. 最小实现查询接口

验收标准：

- 前端可直接消费注册表数据

### B2. 健康聚合（核心）

实现要求（独立）：

- 聚合各子服务 Actuator 健康状态
- 采用短 TTL 缓存，避免频繁探活

测试先行步骤（重点）：

1. 写测试：单服务健康状态映射
2. 写测试：下游超时时返回降级状态
3. 写测试：缓存命中时不重复探活
4. 实现聚合逻辑 + 超时控制 + Caffeine

验收标准：

- 聚合接口在下游异常时仍可稳定响应

### B3. 演示模式管理

实现要求（独立）：

- 支持 Guest Token 签发/吊销
- 只读模式由网关统一限制

测试先行步骤：

1. 写测试：签发 Token 默认带过期时间
2. 写测试：吊销后 Token 立即失效
3. 实现 Token 管理接口

验收标准：

- 演示模式切换立即生效

## 阶段 C：Blog-Service（性能与 SEO）

### C1. 文章与分类模型

实现要求（独立）：

- Post/Tag/Category 多对多关系
- 分页查询、详情查询、基础写操作

测试先行步骤：

1. 写测试：新增文章并绑定标签分类
2. 写测试：分页返回数量与总数正确
3. 最小实现 Repository + Service

验收标准：

- 主流程 CRUD 正常，分页无全量扫描

### C2. Markdown 渲染缓存（核心）

实现要求（独立）：

- Markdown 转 HTML
- 缓存键包含文章版本（如 `id + updateTime`）

测试先行步骤（重点）：

1. 写测试：首次请求触发渲染并缓存
2. 写测试：二次请求命中缓存
3. 写测试：文章更新后缓存失效并重渲染
4. 实现渲染管线 + Caffeine 缓存策略

验收标准：

- 热门文章详情接口有稳定缓存命中率

### C3. 站内搜索

实现要求（独立）：

- MySQL 全文索引搜索
- 关键字查询支持分页

测试先行步骤：

1. 写测试：关键字命中结果排序符合预期
2. 写测试：空关键字返回参数错误
3. 实现搜索 API 与输入校验

验收标准：

- 搜索接口性能与结果稳定性满足基础可用

---

## 阶段 D：Ops-Core（运维核心）

### D1. K8s 状态采集

实现要求（独立）：

- 拉取 Pod/Deployment/Node 状态摘要
- 采集异常状态并标准化输出

测试先行步骤：

1. 写测试：正常状态映射
2. 写测试：异常状态映射（CrashLoopBackOff 等）
3. 实现状态转换层

验收标准：

- 状态接口可稳定输出前端可读结构

### D2. Jenkins 触发流水线（核心）

实现要求（独立）：

- 通过 REST 触发流水线
- 使用异步任务并限制超时与重试

测试先行步骤（重点）：

1. 写测试：触发成功返回任务 ID
2. 写测试：超时后进入失败状态
3. 写测试：重试达到上限后停止
4. 实现异步触发与状态跟踪

验收标准：

- 触发链路具备可观测、可追踪、可重试能力

### D3. 发布诊断聚合（核心中的核心）

实现要求（独立）：

- 输入部署异常事件
- 聚合 Prometheus 指标 + Loki 日志
- 输出诊断报告与建议

测试先行步骤（重点）：

1. 写测试：仅有指标异常时报告结论
2. 写测试：仅有错误日志时报告结论
3. 写测试：指标+日志同时异常时提高严重级别
4. 写测试：外部依赖部分不可用时输出降级报告
5. 实现诊断规则引擎（先最小规则集）
6. 测试通过后再重构为可扩展规则链

验收标准：

- 报告结果具备一致结构、清晰结论、可执行建议

## 5. 每个步骤的交付物清单（统一）

每个独立步骤结束后必须提交以下内容：

- 代码实现（最小可行）
- 对应测试（至少 1 个失败转通过的测试用例）
- 测试执行记录（通过/失败截图或日志摘要）
- 接口文档变更（如有）
- 风险与后续改进项（1~3 条）

## 6. 分支与提交建议

- 分支命名：`feat/<module>-<slice>` 或 `fix/<module>-<slice>`
- 提交粒度：一个提交只包含一个可验证切片
- 推荐提交模板：
  - `test(module): add failing test for <behavior>`
  - `feat(module): implement <behavior> to pass tests`
  - `refactor(module): improve readability with tests passing`

## 7. CI 门禁（TDD 强约束）

流水线最少包含以下门禁：

1. 单元测试必须通过
2. 核心模块覆盖率阈值（建议：common/gateway/ops 不低于 70%）
3. 集成测试主链路通过
4. 禁止跳过测试直接构建发布

## 8. 里程碑建议（按 TDD 产能节奏）

- M1：完成阶段 A + B（规范底座 + 网关核心）
- M2：完成阶段 C（博客主链路 + 渲染缓存）
- M3：完成阶段 D（运维采集 + 诊断引擎）
- M4：联调前端看板 + 全链路回归 + 发布基线

## 9. 执行纪律

- 先写测试，再写实现；任何“先实现后补测”都必须说明原因
- 一个步骤未通过测试，不进入下一步骤
- 核心链路（鉴权、诊断、缓存）必须保留失败用例历史
