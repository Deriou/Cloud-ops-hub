# L1.5：部署实操命令与知识点速查

本文档总结 Cloud-Ops-Hub 从零部署到域名可访问全过程中使用的命令和涉及的核心知识点。

## 一、kubectl 核心命令

### 资源查看

```bash
# 查看 Pod 状态（最常用）
kubectl -n cloud-ops get pod

# 持续监听 Pod 变化（-w = watch）
kubectl -n cloud-ops get pod -w

# 查看 Service 列表
kubectl -n cloud-ops get svc

# 查看 Ingress
kubectl -n cloud-ops get ingress

# 查看 PV/PVC 绑定状态
kubectl get pv
kubectl -n cloud-ops get pvc

# 查看 Secret 和 ConfigMap
kubectl -n cloud-ops get secret,configmap
```

### 排障三件套

```bash
# 1. 查看 Pod 事件链（镜像拉取失败、探针失败、调度失败都在这里）
kubectl -n cloud-ops describe pod <pod-name>

# 2. 查看应用日志
kubectl -n cloud-ops logs deploy/<name> --tail=100

# 3. 查看上一次崩溃的日志（CrashLoopBackOff 时用）
kubectl -n cloud-ops logs deploy/<name> --tail=100 --previous
```

### 资源操作

```bash
# 通过 Kustomize 一键部署所有清单
kubectl apply -k infra/k8s/base/

# 删除 Pod（Deployment 会自动重建）
kubectl -n cloud-ops delete pod <pod-name>

# 创建 Secret（命令式，不入库）
kubectl -n cloud-ops create secret generic cloud-ops-secret \
  --from-literal=OPS_AUTH_MASTER_KEY=xxx \
  --from-literal=BLOG_DATASOURCE_PASSWORD=xxx

# 更新已有 Secret（dry-run + apply 模式）
kubectl -n cloud-ops create secret generic cloud-ops-secret \
  --from-literal=KEY=VALUE \
  --dry-run=client -o yaml | kubectl apply -f -

# 创建 imagePullSecret（私有镜像仓库认证）
kubectl -n cloud-ops create secret docker-registry acr-secret \
  --docker-server=<registry-url> \
  --docker-username=<user> \
  --docker-password=<pass>

# 进入容器执行命令
kubectl -n cloud-ops exec -it deploy/mysql -- mysql -uroot -p
```

### 系统组件排查

```bash
# 查看 kube-system 下的系统 Pod（CoreDNS、Traefik 等）
kubectl -n kube-system get pods

# 查看某个 Pod 使用的镜像
kubectl -n kube-system describe pod <pod-name> | grep Image
```

## 二、Docker 核心命令

### 镜像构建

```bash
# 标准构建（上下文为当前目录）
docker build -t <image:tag> .

# 指定 Dockerfile 路径（mono-repo 场景，上下文仍为项目根目录）
docker build -f apps/gateway-portal/Dockerfile -t <image:tag> .

# 指定目标平台（Mac M-series 构建给 Intel ECS 用）
docker build --platform linux/amd64 -t <image:tag> .
```

### 镜像管理

```bash
# 登录私有仓库
docker login <registry-url>

# 给镜像打标签（重命名）
docker tag <source:tag> <target:tag>

# 推送到远程仓库
docker push <image:tag>

# 拉取镜像
docker pull --platform linux/amd64 <image:tag>

# 导出镜像为 tar（用于跨运行时传递）
docker save <image:tag> -o image.tar

# 管道方式导出并导入 K3s containerd
docker save <image:tag> | sudo ctr -n k8s.io images import -
```

### containerd 命令（K3s 底层）

```bash
# 拉取镜像（K3s 用 crictl，不是 docker）
sudo crictl pull <image:tag>

# 导入镜像到 K3s 的 containerd
sudo ctr -n k8s.io images import image.tar
```

## 三、系统管理命令

### 服务管理

```bash
# 重启 K3s
sudo systemctl restart k3s

# 重启 Docker
sudo systemctl restart docker

# 重新加载 systemd 配置（修改 daemon.json 后）
sudo systemctl daemon-reload
```

### 用户与权限

```bash
# 把用户加入 docker 组（免 sudo 使用 docker）
sudo usermod -aG docker deriou

# 配置 kubectl 免 sudo
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown deriou:deriou ~/.kube/config

# 给目录设置属主（MySQL 数据目录需要 UID 999）
sudo chown 999:999 /opt/cloud-ops/mysql-data
```

### 网络诊断

```bash
# 检查端口监听
sudo ss -tlnp | grep :80

# DNS 解析验证
ping deriou.com
```

## 四、核心知识点总结

### K8s 资源模型

| 资源 | 作用 | 关键点 |
|------|------|--------|
| **Namespace** | 资源隔离 | 所有业务资源统一放 `cloud-ops` |
| **Deployment** | 管理 Pod 副本 | 无状态服务用 RollingUpdate，有状态单实例用 Recreate |
| **Service** | 稳定的访问入口 | ClusterIP 类型提供集群内 DNS，如 `mysql.cloud-ops.svc.cluster.local` |
| **ConfigMap** | 非敏感配置 | 可入库，通过 `envFrom` 注入为环境变量 |
| **Secret** | 敏感信息 | 不入库，只提供 example 模板 |
| **PV/PVC** | 持久化存储 | PV 是集群级存储资源，PVC 是命名空间级的使用申请 |
| **Ingress** | 外部流量路由 | 基于域名+路径分发到不同 Service |

### 持久化存储要点

- PV 是"存储资源"，PVC 是"使用申请"，两者通过 `storageClassName` + `volumeName`/`claimRef` 绑定
- `hostPath` 类型 PV 适合单节点学习，生产用云盘 CSI
- `persistentVolumeReclaimPolicy: Retain` 防止 PVC 删除后数据丢失
- MySQL 容器以 UID 999 运行，hostPath 目录需要 `chown 999:999`
- `/docker-entrypoint-initdb.d/` 中的 SQL 只在数据目录为空时执行一次

### Dockerfile 多阶段构建

```
构建阶段（maven/node）   →   运行阶段（JRE/nginx）
包含编译工具，体积大          只有运行时，体积小
构建完丢弃                   最终镜像
```

- 先复制依赖文件（pom.xml / package.json），再复制源码，利用 Docker layer cache
- mono-repo 中需要从项目根目录构建：`docker build -f apps/xxx/Dockerfile .`
- 生产镜像必须包含 HEALTHCHECK

### 网络与 DNS

- K8s 内部 DNS 由 CoreDNS 提供，Service 名自动注册为 DNS 记录
- 格式：`<service-name>.<namespace>.svc.cluster.local`
- CoreDNS 挂了 = 所有 Service 域名解析失败 = 所有服务间通信断裂
- Traefik 是 K3s 默认 Ingress Controller，监听 80/443 端口，根据 Ingress 规则路由流量

### 镜像管理策略（国内 ECS）

```
Docker Hub 不可达
    ↓
方案 A：Docker daemon 配阿里云镜像加速器（/etc/docker/daemon.json）
    → 适用于 Docker 构建过程
方案 B：K3s containerd 配镜像加速（/etc/rancher/k3s/registries.yaml）
    → 适用于 K3s 系统组件
方案 C：Mac 拉取 → 推到 ACR → ECS 从 ACR 拉取
    → 适用于加速器也没缓存的镜像（rancher/* 系列）
```

关键配置文件：
- `/etc/docker/daemon.json` — Docker 引擎配置（镜像加速器）
- `~/.docker/config.json` — Docker 登录凭证（不要和上面搞混）
- `/etc/rancher/k3s/registries.yaml` — K3s containerd 镜像源

### Kustomize

- `kustomization.yaml` 是清单目录，列出所有要部署的 YAML 文件
- `kubectl apply -k <dir>` 一键部署目录下所有资源
- `base/` 放通用配置，`overlays/dev/` 放环境差异覆盖
- 不在列表里的文件不会被部署

### 部署流程（固化为习惯）

```
本地开发改代码/清单
    ↓
git commit + push
    ↓
ECS: git pull
    ↓
需要重建镜像？ → docker build + docker push (ACR)
    ↓
kubectl apply -k infra/k8s/base/
    ↓
kubectl -n cloud-ops get pod（验证）
    ↓
出问题 → describe / logs / logs --previous
```
